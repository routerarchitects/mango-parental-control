# Config-Raw Generation Rules

## 1. Purpose

This document defines how Mango Parental Control Service generates parental-control-owned `config-raw`.

It defines:

- ownership and boundaries
- response behavior
- command tuple format
- effective policy model
- full-snapshot generation
- rendering templates
- deterministic ordering
- snapshot comparison
- generation algorithm
- examples
- policy hash inputs

Identifiers such as `g001`, `s001`, and sample labels such as `MAC X` or `MAC Y` are symbolic placeholders in explanatory examples unless a concrete MAC address is shown.

---

## 2. Ownership And Boundaries

### 2.1 Generated Scope

Mango Parental Control Service shall generate only parental-control-owned `config-raw`.

The service shall generate command tuples only for configuration sections owned by parental-control policy rendering.

### 2.2 Not Generated By This Service

The service shall not generate or modify non-parental-control configuration sections.

The service shall not execute gateway commands. It only returns command tuples in `config-raw` format.

Gateway apply actions such as commit/restart are owned by the downstream apply flow.

The service shall not emit:

```text
uci commit firewall
uci commit dhcp
/etc/init.d/firewall restart
/etc/init.d/dnsmasq restart
```

### 2.3 Upstream Schema Reference

The upstream `config-raw` schema shall be used as the command-shape reference:

`https://github.com/Telecominfraproject/wlan-ucentral-schema/blob/main/schema/config-raw.yml`

---

## 3. Response Contract

### 3.1 Successful Write Response

Successful write responses shall return `200 OK`.

The service intentionally standardizes successful create, update, and delete operations on `200 OK` instead of using `201 Created` for create operations.

If device-side enforcement changes, the response shall include `config-raw` containing the full parental-control-owned snapshot after the write. If the resulting snapshot is empty, `config-raw` shall contain an empty array `[]`.

### 3.2 When `config-raw` Is Omitted

The response body shall not include `config-raw` when:

- device-side enforcement does not change
- the request fails validation or cannot be processed

Failed requests shall return an error response instead of `config-raw`.

### 3.3 Empty Snapshot Rule

When the resulting parental-control-owned snapshot is empty (no active parental-control commands remain), the response body shall include `config-raw` as an empty array `[]` to signal that all parental-control-owned sections should be cleared on the device.

---

## 4. Command Tuple Contract

### 4.1 Tuple Format

The generated payload shall use the existing `config-raw` array model.

Each command tuple shall contain:

```text
[command, path]
[command, path, value]
```

Examples:

```json
["set", "firewall.pc_rule_g001_s003_internet", "rule"]
```

```json
["set", "firewall.pc_rule_g001_s003_internet.target", "REJECT"]
```

```json
["add_list", "firewall.pc_rule_g001_s003_internet.src_mac", "B4:6A:D4:45:E9:5C"]
```

### 4.2 Supported Commands

 parental-control rendering shall use:

- `set`
- `add_list`

### 4.3 Unsupported Commands

 parental-control rendering shall not use explicit delete commands for parental-control-owned sections.

Missing parental-control-owned sections are removed by the downstream device-side apply flow when it applies the full snapshot.

---

## 5. Effective Policy Model

### 5.1 Required Stored Inputs

For one subscriber, the renderer shall read:

- all groups with `group_config_index`
- all group devices
- all schedules with `schedule_config_index`
- all group-schedule links

The renderer shall derive the effective active relations from stored state.

The renderer shall not depend on previously applied gateway configuration.

### 5.2 Active Relation Filter

One group-schedule relation is active only when all of the following are true:

```text
group exists
group has at least one device
group-schedule link exists
schedule exists
schedule.enabled = true
```

Inactive relations shall not contribute any commands to the rendered snapshot.

### 5.3 Fields That Affect Rendering

These inputs affect generated config:

```text
group.group_config_index
schedule.schedule_config_index
device MAC list
schedule.enabled
schedule.action_type
schedule.target_kind
schedule.target_value
schedule.start_minute
schedule.stop_minute
schedule.weekdays
group-schedule link existence
```

### 5.4 Fields That Do Not Affect Rendering

These inputs do not affect generated config:

```text
group.name
group.description
schedule.name
schedule.description
created_at
updated_at
```

Changing only display metadata shall not require config generation.

---

## 6. Full Snapshot Rule

### 6.1 Full Snapshot Requirement

The service shall return a full snapshot of all parental-control-owned `config-raw` after each write that changes effective device-side enforcement.

Every active parental-control-owned section required after the write shall be returned.

If the effective parental-control snapshot contains multiple active parental-control sections, the returned `config-raw` shall contain all active parental-control-owned sections in that snapshot.

### 6.2 Partial Payload Behavior

The returned payload shall not contain only one section from the effective parental-control snapshot.

If a partial payload is returned, omitted parental-control-owned sections may be removed on the device by the downstream apply flow.

### 6.3 Empty Snapshot Behavior

If the resulting parental-control-owned snapshot is empty, the response body shall include `config-raw` as an empty array `[]`.

---

## 7. Rendering Rules

### 7.1 Value Transformations

The renderer shall transform normalized stored values as follows.

`start_minute` to `start_time`:

- render in `HH:MM:SS`
- use zero-padded 24-hour format
- `1260` renders as `21:00:00`
- `540` renders as `09:00:00`

`stop_minute` to `stop_time`:

- render in `HH:MM:SS`
- use zero-padded 24-hour format

`weekdays` array to `weekdays` string:

- map `0..6` to `Sun Mon Tue Wed Thu Fri Sat`
- preserve ascending weekday order in the rendered string
- join with one space
- `[1, 2, 3, 4, 5]` renders as `Mon Tue Wed Thu Fri`
- `[0, 1, 2, 3, 4, 5, 6]` renders as `Sun Mon Tue Wed Thu Fri Sat`

`client_mac` values:

- normalize to uppercase colon-separated MAC representation before ordering and rendering
- emit using `add_list`

### 7.1.1 Deployment Defaults For Fixed Rule Fields

Some rendered firewall fields are deployment defaults rather than subscriber-owned data.

These values shall come from deployment configuration with file-backed defaults.

Default file:

`configs/defaults/config-raw-defaults.yaml`

Default values:

```yaml
firewall_defaults:
 src: down1v0
 dest: up0v0
 family: any
 proto: all
 target: REJECT
```

Recommended environment overrides:

```text
PC_RENDER_DEFAULTS_FILE
PC_FIREWALL_DEFAULT_SRC
PC_FIREWALL_DEFAULT_DEST
PC_FIREWALL_DEFAULT_FAMILY
PC_FIREWALL_DEFAULT_PROTO
PC_FIREWALL_DEFAULT_TARGET
```

 precedence shall be:

1. explicit deployment environment override
2. defaults file value

These fields shall not be client-updateable through the parental-control API.

### 7.2 Section Naming

Generated section names shall be deterministic and stable.

They shall be based on stable internal config indexes, not display names.

Indexes shall be rendered as zero-padded three-digit decimal values.

For `target_kind = INTERNET`:

```text
firewall.pc_rule_g<group_config_index_3d>_s<schedule_config_index_3d>_internet
```

The rendered display name shall be:

```text
PC_Block_g<group_config_index_3d>_s<schedule_config_index_3d>_Internet
```

For `target_kind = APP` and `target_value = YOUTUBE`:

```text
dhcp.pc_app_youtube_domains
firewall.pc_app_youtube_ipset
firewall.pc_rule_g<group_config_index_3d>_s<schedule_config_index_3d>_youtube
```

The rendered firewall rule display name shall be:

```text
PC_Block_g<group_config_index_3d>_s<schedule_config_index_3d>_YouTube
```

The service shall not use mutable display fields in section names:

```text
group.name
schedule.name
description
```

### 7.3 Command Ordering

The renderer shall emit commands in deterministic order:

1. shared sections before per-group firewall rules
2. groups in `group_config_index` ascending order
3. schedules in `schedule_config_index` ascending order inside each group
4. parent section creation before section fields
5. scalar `set` commands before `add_list` commands for the same section
6. repeated list values in normalized ascending order

Within a single section, the scalar `set` commands shall be ordered consistently across all templates:
1. `name`
2. `enabled`
3. Network fields (`src`, `dest`, `family`, `proto`, `ipset`, `target`) in alphabetical order of their property names
4. Time fields (`start_time`, `stop_time`)
5. Weekdays (`weekdays`)

### 7.4 MAC Ordering

MAC lists inside a rendered rule shall be emitted in normalized ascending MAC order.

## 8. INTERNET Block Template

### 8.1 Supported Input

For:

```text
action_type = BLOCK
target_kind = INTERNET
target_value = null
```

the renderer shall emit exactly one firewall rule section per active group-schedule relation.

### 8.2 Section Names

Section example:

```text
firewall.pc_rule_g001_s003_internet
```

Display name example:

```text
PC_Block_g001_s003_Internet
```

### 8.3 Fixed Fields

Fixed rendered field values:

| Field | Rendered value |
|---|---|
| section type | `rule` |
| `enabled` | `"1"` |
| `dest` | deployment default, defaulting to `"up0v0"` |
| `family` | deployment default, defaulting to `"any"` |
| `proto` | deployment default, defaulting to `"all"` |
| `src` | deployment default, defaulting to `"down1v0"` |
| `target` | deployment default, defaulting to `"REJECT"` |

### 8.4 Variable Fields

Variable rendered field values:

| Field | Source |
|---|---|
| section name | group and schedule config indexes |
| display `name` | group and schedule config indexes |
| `start_time` | `start_minute` |
| `stop_time` | `stop_minute` |
| `weekdays` | `weekdays[]` |
| `src_mac` | all normalized devices in the group |

### 8.5 Command Order

For one INTERNET rule section, the command order shall be:

1. section create command
2. `name`
3. `enabled`
4. `dest`
5. `family`
6. `proto`
7. `src`
8. `target`
9. `start_time`
10. `stop_time`
11. `weekdays`
12. `src_mac` values in normalized ascending order

### 8.6 Example

```json
{
 "config-raw": [
 ["set", "firewall.pc_rule_g001_s003_internet", "rule"],
 ["set", "firewall.pc_rule_g001_s003_internet.name", "PC_Block_g001_s003_Internet"],
 ["set", "firewall.pc_rule_g001_s003_internet.enabled", "1"],
 ["set", "firewall.pc_rule_g001_s003_internet.dest", "up0v0"],
 ["set", "firewall.pc_rule_g001_s003_internet.family", "any"],
 ["set", "firewall.pc_rule_g001_s003_internet.proto", "all"],
 ["set", "firewall.pc_rule_g001_s003_internet.src", "down1v0"],
 ["set", "firewall.pc_rule_g001_s003_internet.target", "REJECT"],
 ["set", "firewall.pc_rule_g001_s003_internet.start_time", "09:00:00"],
 ["set", "firewall.pc_rule_g001_s003_internet.stop_time", "17:00:00"],
 ["set", "firewall.pc_rule_g001_s003_internet.weekdays", "Sun Mon Tue Wed Thu Fri Sat"],
 ["add_list", "firewall.pc_rule_g001_s003_internet.src_mac", "1A:F3:33:86:97:0A"],
 ["add_list", "firewall.pc_rule_g001_s003_internet.src_mac", "B4:6A:D4:45:E9:5C"]
 ]
}
```

## 9. APP / YouTube Block Template

### 9.1 Supported Input

For:

```text
action_type = BLOCK
target_kind = APP
target_value = YOUTUBE
```

the renderer shall emit APP/YOUTUBE blocking config.

### 9.2 Shared Sections

The renderer shall emit:

```text
dhcp.pc_app_youtube_domains
firewall.pc_app_youtube_ipset
```

Shared YouTube sections shall be emitted once per subscriber snapshot, not once per group.

### 9.3 Firewall Rule Section

The renderer shall emit one firewall rule section per active group-schedule relation using the YouTube target:

```text
firewall.pc_rule_g<group_config_index_3d>_s<schedule_config_index_3d>_youtube
```

The group firewall rule shall include:

- section type `rule`
- deterministic rule name from config indexes
- `enabled` set to `"1"`
- `src` resolved from deployment defaults
- `dest` resolved from deployment defaults
- `target` resolved from deployment defaults
- target-specific `ipset`
- transformed time window fields
- transformed weekdays field
- all normalized group device MACs

In, the deployment defaults for the examples above shall resolve to:

- `src = "down1v0"`
- `dest = "up0v0"`
- `target = "REJECT"`

### 9.4 Shared Section Lifecycle

When shared target-level sections are used, their lifecycle shall be computed against the effective subscriber-wide APP policy, not only against one group-schedule relation.

- the service shall recompute whether shared APP sections are still required across all active APP rules for the subscriber before emitting `config-raw`
- shared APP sections shall be present in the returned snapshot whenever any active APP rule for the subscriber depends on them
- if no active APP rule for the subscriber depends on the shared sections, they shall be absent from the returned snapshot
- the returned snapshot shall therefore be based on the full rendered subscriber policy, not on isolated per-link mutation only

## 10. Snapshot Comparison

### 10.1 Rendered Snapshot Comparison

After rendering the full parental-control-owned snapshot for the subscriber, the service shall compare:

```text
previous rendered snapshot
new rendered snapshot
```

The comparison shall use the fully ordered rendered command list.

### 10.2 Response Decision

If the newly rendered snapshot is identical to the previously rendered snapshot, the successful response body shall not include `config-raw`.

If the newly rendered snapshot differs, the successful response body shall include `config-raw`.

---

## 11. Generation Algorithm

### 11.1 Pseudocode

```text
load subscriber groups
load subscriber group devices
load subscriber schedules
load subscriber group-schedule links

build active relations where:
 group exists
 group has devices
 link exists
 schedule exists
 schedule.enabled = true

render shared APP sections required by active APP relations

for each active group ordered by group_config_index:
 normalize and sort group MAC list
 for each active schedule linked to the group ordered by schedule_config_index:
 if target_kind = INTERNET:
 render one INTERNET firewall rule section
 if target_kind = APP and target_value = YOUTUBE:
 render one YOUTUBE firewall rule section

sort commands using deterministic command ordering rules

compare rendered snapshot with previous rendered snapshot

if unchanged:
 omit config-raw
else:
 return config-raw
```

```json
 ["set", "firewall.pc_rule_g003_s003_internet.dest", "up0v0"],
 ["set", "firewall.pc_rule_g003_s003_internet.family", "any"],
 ["set", "firewall.pc_rule_g003_s003_internet.proto", "all"],
 ["set", "firewall.pc_rule_g003_s003_internet.src", "down1v0"],
 ["set", "firewall.pc_rule_g003_s003_internet.target", "REJECT"],
 ["set", "firewall.pc_rule_g003_s003_internet.start_time", "21:00:00"],
 ["set", "firewall.pc_rule_g003_s003_internet.stop_time", "09:00:00"],
 ["set", "firewall.pc_rule_g003_s003_internet.weekdays", "Sun Mon Tue Wed Thu Fri Sat"],
 ["add_list", "firewall.pc_rule_g003_s003_internet.src_mac", "MAC X"],
 ["add_list", "firewall.pc_rule_g003_s003_internet.src_mac", "MAC Y"]
 ]
}
```

### 12.2 Multiple Active Rules

Effective parental-control snapshot:

```text
firewall.pc_rule_g001_s001_internet
firewall.pc_rule_g002_s001_internet
firewall.pc_rule_g003_s001_internet
```

The returned `config-raw` shall contain all 3 active rules.

Illustrative shape:

```json
{
 "config-raw": [
 ["set", "firewall.pc_rule_g001_s001_internet", "rule"],
 ["set", "firewall.pc_rule_g001_s001_internet.name", "PC_Block_g001_s001_Internet"],

 ["set", "firewall.pc_rule_g002_s001_internet", "rule"],
 ["set", "firewall.pc_rule_g002_s001_internet.name", "PC_Block_g002_s001_Internet"],

 ["set", "firewall.pc_rule_g003_s001_internet", "rule"],
 ["set", "firewall.pc_rule_g003_s001_internet.name", "PC_Block_g003_s001_Internet"]
 ]
}
```

### 12.3 Removing One Active Rule While Others Remain

If the effective parental-control snapshot contains only:

```text
firewall.pc_rule_g001_s001_internet
```

the returned `config-raw` shall contain only that remaining active rule.

No delete command shall be emitted for inactive rules.

### 12.4 No Active Rules Remaining

If no active parental-control sections remain after the write (and active rules previously existed), the successful response body shall include `config-raw` as an empty array `[]` to signal that all parental-control-owned sections should be cleared on the device.

Example:
```json
{
 "config-raw": []
}
```

## 13. Policy Hash

The service may calculate a normalized effective policy hash per subscriber.

The hash shall include enforcement inputs only.

Hash inputs include:

```text
group.group_config_index
schedule.schedule_config_index
schedule.enabled
schedule.target_kind
schedule.target_value
device MAC list
start_minute
stop_minute
weekdays
group-schedule link existence
```

Hash inputs shall not include:

```text
group.name
group.description
schedule.name
schedule.description
created_at
updated_at
```

If normalized effective policy changes:

```text
policy_hash changes
config-raw contains the full rendered parental-control snapshot
```

If normalized effective policy does not change:

```text
policy_hash remains unchanged
response body does not include config-raw
```
